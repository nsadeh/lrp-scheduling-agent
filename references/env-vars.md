# Environment Variables Reference

Every env var read by the codebase, with file:line citations, defaults, and current dev status.

Generated by auditing every `os.environ.get`, `os.environ[...]`, and `getenv` call in [services/api/src](../services/api/src) and [services/api/scripts](../services/api/scripts), then cross-referencing the developer's local [services/api/.env](../services/api/.env).

## Status legend

- **Set in dev**: present in `services/api/.env` and consumed by the code
- **Code-default**: not in `.env`; code relies on the hard-coded default
- **Unused in dev**: in `.env` but never read by any code (candidate for removal)
- **Ambient**: injected by the platform (Railway, Docker) — not in `.env` by design

- **Read in**: [api/gmail/webhook.py:27](../services/api/src/api/gmail/webhook.py:27).

- **Purpose**: Email of the service account expected to sign Pub/Sub push JWTs. The webhook rejects requests signed by any other identity.
- **Status**: Set in dev — `pubsub-push@ai-agents-dev-492713.iam.gserviceaccount.com`.
- **Read in**: [api/gmail/webhook.py:23](../services/api/src/api/gmail/webhook.py:23).
- **Default fallback**: `gmail-api-push@system.gserviceaccount.com` (the Google-managed default for Gmail push, used in most vanilla setups).

## Loader wiring: local vs Railway

Two different mechanisms populate `os.environ` depending on where the code runs:

### Local development
`.env` is only auto-loaded by entrypoints that explicitly call `python-dotenv`:

- [api/main.py:8](../services/api/src/api/main.py:8) — the FastAPI app loads `.env` via `load_dotenv()`.
- Every `scripts/test_*.py` and `scripts/gmail_oauth.py` — CLI tooling does the same.

The arq worker entrypoint [api/gmail/workers.py](../services/api/src/api/gmail/workers.py) does **not** call `load_dotenv` in Python. Instead, [scripts/dev-worker.sh:6-8](../scripts/dev-worker.sh:6) sources `.env` into the shell (`set -a; source .env; set +a`) before launching `arq`, so every var in `.env` becomes a real shell export visible to the worker process.

### Railway (staging & production)
Railway injects env vars directly into each service's container process — **there is no `.env` file on Railway and `load_dotenv()` is a no-op in that environment.** Each Railway service (api, worker, etc.) has its own independent env var set, configured via the Railway dashboard or `railway variables`.

That means adding a worker service on Railway requires copying the full set of app env vars to it (or using Railway's `${{Redis.REDIS_URL}}`-style reference variables for shared ones). See [railway-deployment.md](railway-deployment.md) if/when that runbook gets written.
